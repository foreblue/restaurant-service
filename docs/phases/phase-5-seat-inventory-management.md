# Phase 5. 좌석/재고 관리

## 개요

Phase 5는 Phase 1-4 이후의 확장 단계로, 매장이 이미 입점, 예약 페이지, 기본 예약, 오너 예약 관리, 예약금/환불 흐름을 갖춘 상태에서 시간대별 수용량을 더 정교하게 통제하기 위한 단계다.

기준 문서에서 MVP의 최소 완성 지점은 Phase 1-4로 정의되어 있다. 따라서 Phase 5는 예약 기능 자체를 처음 완성하는 범위가 아니라, 예약 가능 여부를 단순 상품 재고 수량에서 실제 좌석, 테이블, 이용 시간, 타임슬롯 단위 재고로 고도화하는 범위다.

이 단계의 핵심은 고객이 특정 날짜, 시간, 인원, 예약 상품을 선택했을 때 실제로 수용 가능한 좌석 재고가 있는지 계산하고, 예약 확정 또는 결제 대기 중 중복 예약이 발생하지 않도록 보호하는 것이다.

## 목표와 비목표

### 목표

- 매장 오너가 예약에 사용할 테이블을 등록하고 운영 상태를 관리한다.
- 테이블별 좌석 유형, 최소/최대 수용 인원, 운영 가능 상품을 설정한다.
- 예약 상품별 기본 이용 시간을 기준으로 타임슬롯 재고를 생성한다.
- 예약 생성, 변경, 취소, 결제 실패, 매장 취소에 따라 재고를 차감하거나 복구한다.
- 동일 테이블 또는 동일 타임슬롯에 중복 예약이 들어가지 않도록 API와 데이터베이스 수준에서 방지한다.
- 매장이 특정 날짜 또는 시간대의 좌석을 임시 마감할 수 있게 한다.
- 기존 Phase 2-4 예약, 결제, 환불 흐름과 충돌하지 않는 재고 계산 규칙을 정의한다.

### 비목표

- 단체 예약, 대형 단체, 대관, 별도 견적 기반 예약은 다루지 않는다.
- POS 연동을 통한 실시간 착석, 주문, 결제 상태 동기화는 다루지 않는다.
- 직원별 권한 분리나 테이블 담당자 배정은 다루지 않는다.
- 웨이팅, 지도 기반 탐색, 통합 검색 기능은 다루지 않는다.
- 복잡한 플로어 맵 에디터, 드래그 앤 드롭 좌석 배치도, 실시간 홀 운영 보드는 이 단계의 필수 범위가 아니다.
- 테이블 조합 자동 최적화는 MVP 이후 제한적으로 다루며, Phase 5에서는 수동 등록된 제한적 조합만 고려한다.

## 사용자 역할

| 역할 | Phase 5에서의 사용 범위 |
| --- | --- |
| 고객 | 예약 페이지에서 상품, 날짜, 시간, 인원을 선택하고 실제 재고가 있는 시간만 예약한다. |
| 매장 오너 | 테이블, 좌석 유형, 이용 시간, 상품별 좌석 연결, 타임슬롯 재고, 임시 마감을 관리한다. |
| 플랫폼 관리자 | 매장별 재고 설정 오류, 중복 예약 예외, 운영 문의를 확인하고 필요 시 데이터를 보정한다. |

MVP 기준과 동일하게 매장 직원 권한 분리는 제공하지 않는다. 매장 오너가 모든 좌석/재고 설정을 관리하는 단일 운영자 모델로 본다.

## 좌석/재고 운영 시나리오

### 1. 최초 좌석 설정

1. 매장 오너가 테이블을 등록한다.
2. 테이블명, 좌석 유형, 최소/최대 수용 인원, 사용 여부를 입력한다.
3. 예약 상품별로 사용 가능한 좌석 유형 또는 개별 테이블을 연결한다.
4. 상품별 기본 이용 시간과 예약 가능 시간 정책을 확인한다.
5. 시스템이 영업시간, 브레이크타임, 휴무, 상품 정책을 기준으로 타임슬롯 후보를 생성한다.

### 2. 고객 예약 가능 시간 조회

1. 고객이 예약 상품과 날짜를 선택한다.
2. 시스템이 해당 상품의 예약 가능 요일, 예약 가능 시간, 예약 오픈 기간, 예약 마감 시간을 확인한다.
3. 고객이 인원을 선택하면 수용 가능한 테이블 후보를 찾는다.
4. 각 시간대별로 이미 점유된 예약과 임시 마감 슬롯을 제외한다.
5. 예약 가능한 시간만 고객에게 노출한다.

### 3. 예약 생성과 재고 차감

1. 고객이 시간과 인원을 선택하고 예약 생성을 요청한다.
2. 시스템이 트랜잭션 안에서 대상 타임슬롯과 테이블 후보를 잠근다.
3. 결제 또는 카드 보증이 필요한 상품이면 `pending` 예약도 일정 시간 재고를 점유한다.
4. 예약이 확정되면 해당 시간 범위의 재고가 차감된다.
5. 결제 실패, 제한 시간 초과, 고객 취소, 매장 취소 시 재고를 복구한다.

### 4. 예약 변경

1. 오너 또는 고객이 예약 날짜, 시간, 인원 변경을 요청한다.
2. 변경 후 조건으로 재고를 다시 계산한다.
3. 새 슬롯 확보가 성공하면 기존 슬롯 점유를 해제하고 새 슬롯을 점유한다.
4. 새 슬롯 확보가 실패하면 기존 예약은 유지한다.

### 5. 임시 마감

1. 매장 오너가 특정 날짜, 시간대, 상품, 좌석 유형 또는 테이블을 선택해 임시 마감한다.
2. 시스템은 해당 범위의 신규 예약을 차단한다.
3. 이미 확정된 예약은 자동 취소하지 않고 오너가 별도로 변경 또는 취소 처리한다.
4. 임시 마감 해제 시 이후 예약 가능 조회에 다시 반영한다.

## 기능 상세

### 테이블 등록

- 매장 단위로 예약에 사용할 테이블을 등록한다.
- 필수 입력값은 테이블명, 좌석 유형, 최소 수용 인원, 최대 수용 인원, 사용 여부다.
- 선택 입력값은 설명, 정렬 순서, 내부 메모, 운영 제외 기간이다.
- 테이블 삭제는 예약 이력이 없는 경우에만 허용한다.
- 예약 이력이 있는 테이블은 비활성화 처리만 허용해 과거 예약 기록과 연결을 유지한다.

### 좌석 유형

- 좌석 유형은 고객 노출과 재고 분류에 사용한다.
- 기본 유형 예시는 다음과 같다.

| 유형 | 설명 |
| --- | --- |
| hall | 일반 홀 좌석 |
| room | 룸 좌석 |
| counter | 바 또는 카운터 좌석 |
| terrace | 테라스 좌석 |

- 좌석 유형 이름은 매장별 표시명으로 커스터마이즈할 수 있다.
- 좌석 유형은 예약 상품과 연결되어 특정 상품이 특정 좌석에서만 예약되도록 제한할 수 있다.

### 수용 인원

- 테이블은 `min_party_size`와 `max_party_size`를 가진다.
- 고객 예약 인원은 테이블의 최소/최대 수용 인원 범위 안에 있어야 한다.
- 상품 자체의 최소/최대 인원과 테이블 수용 인원을 모두 만족해야 한다.
- 기준 문서의 MVP 제외 범위에 따라 일반 예약 중심으로 보며, 예시는 1-8인 범위를 기본 상한으로 둔다.
- 매장 전체 최대 예약 인원은 상품, 테이블, 매장 정책 중 가장 엄격한 값으로 제한한다.

### 테이블 조합 범위

- 테이블 조합은 MVP 이후 제한적으로 지원한다.
- Phase 5 문서상 초안은 수동으로 등록된 조합만 고려한다.
- 조합은 2개 이상의 테이블을 하나의 예약 후보로 묶는 방식이다.
- 대형 단체, 대관, 별도 견적 예약을 위한 조합 자동 최적화는 제외한다.
- 조합 사용 시 포함된 개별 테이블 중 하나라도 같은 시간대에 점유되어 있으면 조합 전체를 사용할 수 없다.
- 조합은 일반 예약 범위 안에서 인접 테이블을 붙이는 정도로 제한한다.

### 상품별 좌석 연결

- `reservation_products`는 예약 가능한 좌석 유형 또는 개별 테이블과 연결될 수 있다.
- 예시는 다음과 같다.

| 상품 | 연결 예시 |
| --- | --- |
| 일반 좌석 예약 | hall, counter |
| 룸 예약 | room |
| 오마카세 | counter |
| 런치 코스 | hall, room |

- 상품에 좌석 연결이 없으면 매장 내 활성 테이블 전체를 후보로 본다.
- 상품에 좌석 유형과 개별 테이블이 모두 연결된 경우 개별 테이블 제한을 우선한다.
- 상품의 예약 가능 요일, 시간, 인원 정책도 함께 만족해야 한다.

### 기본 이용 시간

- 기본 이용 시간은 예약이 테이블을 점유하는 시간 길이다.
- 매장 기본값과 상품별 오버라이드 값을 둘 수 있다.
- 상품별 값이 있으면 상품별 값을 우선한다.
- 예시는 다음과 같다.

| 상품 | 기본 이용 시간 |
| --- | --- |
| 일반 좌석 예약 | 90분 |
| 런치 코스 | 120분 |
| 디너 코스 | 150분 |
| 룸 예약 | 120분 |

- 예약 변경 시 이용 시간이 달라지는 상품으로 변경하면 점유 시간 범위를 다시 계산한다.

### 타임슬롯 생성

- 타임슬롯은 매장 영업시간, 브레이크타임, 정기 휴무, 임시 휴무, 상품별 예약 가능 시간, 슬롯 간격을 기준으로 생성한다.
- 기준 문서의 MVP 설정과 맞춰 30분 또는 60분 단위를 기본 슬롯 간격으로 둔다.
- 생성 방식은 두 가지를 지원한다.

| 방식 | 설명 |
| --- | --- |
| 동적 계산 | 고객 조회 시 정책과 예약 현황을 기반으로 실시간 계산한다. |
| 사전 생성 | 예약 오픈 기간만큼 `time_slots`를 미리 생성하고 상태를 갱신한다. |

- Phase 5 초안에서는 운영자 화면과 임시 마감을 단순화하기 위해 사전 생성을 기본으로 한다.
- 영업시간 또는 상품 정책이 변경되면 미래 타임슬롯을 재생성하거나 보정한다.
- 이미 예약이 있는 과거 또는 현재 슬롯은 자동 삭제하지 않는다.

### 재고 차감/복구

- 재고 차감은 예약이 특정 테이블 또는 테이블 조합을 점유할 때 발생한다.
- `pending`, `confirmed`, `modified` 상태의 유효 예약은 재고를 점유한다.
- 결제 제한 시간 초과, 결제 실패, 고객 취소, 매장 취소는 재고를 복구한다.
- `completed`, `no_show` 상태는 과거 이력으로 유지되며 신규 예약 가능 계산에서는 현재 시점 이후 슬롯에만 영향을 준다.
- 예약 변경은 새 재고 확보와 기존 재고 해제를 하나의 트랜잭션으로 처리한다.

### 중복 예약 방지

- 동일 테이블이 겹치는 시간 범위에 두 개 이상의 유효 예약을 가질 수 없다.
- 동일 테이블 조합에 포함된 개별 테이블도 각각 중복 점유를 금지한다.
- API 수준에서 가용성 검사를 수행하고, 데이터베이스 수준에서 락과 제약 조건으로 최종 방어한다.
- 예약 생성 요청은 멱등키를 받아 네트워크 재시도에 따른 중복 예약 생성을 방지한다.
- 결제 연동 중 `pending` 예약이 생성되는 경우에도 해당 시간대 재고를 임시 점유한다.

### 임시 마감

- 매장 오너는 날짜, 시간대, 상품, 좌석 유형, 개별 테이블 범위로 임시 마감을 설정할 수 있다.
- 임시 마감 사유는 내부 운영 메모로 저장한다.
- 임시 마감은 신규 예약 가능 조회와 수동 예약 등록 모두에 적용한다.
- 플랫폼 관리자는 분쟁 대응 또는 장애 복구 목적으로 임시 마감 이력을 조회할 수 있다.

- `max_party_size`는 `min_party_size` 이상이어야 한다.

## API 초안

### 테이블 관리

`GET /owner/restaurants/{restaurantId}/tables`

- 설명: 매장 테이블 목록을 조회한다.
- 응답 요약: 테이블 ID, 이름, 좌석 유형, 수용 인원, 활성 여부, 예약 이력 존재 여부.

`POST /owner/restaurants/{restaurantId}/tables`

- 설명: 테이블을 등록한다.
- 요청 요약: `name`, `seatType`, `seatTypeLabel`, `minPartySize`, `maxPartySize`, `isActive`.
- 응답 요약: 생성된 테이블 정보.

`PATCH /owner/restaurants/{restaurantId}/tables/{tableId}`

- 설명: 테이블 정보를 수정한다.
- 요청 요약: 이름, 좌석 유형, 수용 인원, 활성 여부, 정렬 순서.
- 응답 요약: 수정된 테이블 정보.

`DELETE /owner/restaurants/{restaurantId}/tables/{tableId}`

- 설명: 예약 이력이 없는 테이블을 삭제한다.
- 응답 요약: 삭제 결과. 예약 이력이 있으면 비활성화 안내 오류를 반환한다.

### 상품별 좌석 연결

`PUT /owner/restaurants/{restaurantId}/reservation-products/{productId}/seat-rules`

- 설명: 예약 상품에 허용 좌석 유형 또는 테이블을 연결한다.
- 요청 요약: `allowedSeatTypes`, `allowedTableIds`, `defaultDurationMinutes`, `slotIntervalMinutes`.
- 응답 요약: 갱신된 상품 좌석/재고 설정.

`GET /owner/restaurants/{restaurantId}/reservation-products/{productId}/seat-rules`

- 설명: 상품별 좌석 연결과 이용 시간 설정을 조회한다.
- 응답 요약: 허용 좌석 유형, 허용 테이블, 기본 이용 시간, 슬롯 간격.

### 타임슬롯과 재고

`POST /owner/restaurants/{restaurantId}/time-slots/generate`

- 설명: 예약 오픈 기간의 미래 타임슬롯을 생성 또는 보정한다.
- 요청 요약: `fromDate`, `toDate`, `productIds`, `regenerateFutureOnly`.
- 응답 요약: 생성 수, 갱신 수, 제외 수, 오류 목록.

`GET /owner/restaurants/{restaurantId}/time-slots`

- 설명: 오너용 타임슬롯 재고를 조회한다.
- 요청 요약: `date`, `productId`, `seatType`, `includeClosed`.
- 응답 요약: 시간대별 전체 후보, 예약 점유, 남은 재고, 임시 마감 여부.

`GET /public/restaurants/{restaurantId}/availability`

- 설명: 고객 예약 페이지에서 예약 가능한 시간을 조회한다.
- 요청 요약: `productId`, `date`, `partySize`.
- 응답 요약: 예약 가능한 시작 시간, 종료 예상 시간, 남은 재고 표시 여부.

### 예약 생성과 재고 점유

`POST /public/restaurants/{restaurantId}/reservations`

- 설명: 고객 예약을 생성하고 재고를 점유한다.
- 요청 요약: `productId`, `startsAt`, `partySize`, 고객 정보, 요청사항, `idempotencyKey`.
- 응답 요약: 예약 ID, 상태, 결제 필요 여부, 재고 점유 만료 시각.

`PATCH /owner/restaurants/{restaurantId}/reservations/{reservationId}/time`

- 설명: 오너가 예약 시간 또는 인원을 변경한다.
- 요청 요약: `startsAt`, `partySize`, `productId`.
- 응답 요약: 변경된 예약 시간, 배정 테이블, 재고 변경 결과.

### 임시 마감

`POST /owner/restaurants/{restaurantId}/time-slots/closures`

- 설명: 특정 범위의 타임슬롯을 임시 마감한다.
- 요청 요약: `date`, `startsAt`, `endsAt`, `productId`, `seatType`, `tableId`, `reason`.
- 응답 요약: 마감된 슬롯 목록.

`DELETE /owner/restaurants/{restaurantId}/time-slots/closures/{closureId}`

- 설명: 임시 마감을 해제한다.
- 응답 요약: 해제 결과와 다시 예약 가능한 슬롯 목록.

## 재고 계산 규칙

### 기본 후보 계산

1. 매장 영업일, 영업시간, 브레이크타임, 휴무를 확인한다.
2. 상품의 예약 가능 요일과 시간 범위를 확인한다.
3. 예약 오픈 기간과 예약 마감 시간을 확인한다.
4. 고객 인원이 상품 인원 범위와 매장 인원 범위를 만족하는지 확인한다.
5. 상품에 연결된 좌석 유형 또는 테이블 제한을 적용한다.
6. 고객 인원을 수용할 수 있는 테이블 또는 제한적 조합 후보를 찾는다.
7. 예약 시작 시각부터 이용 시간 종료 시각까지 겹치는 기존 점유를 제외한다.
8. 임시 마감된 슬롯 또는 테이블을 제외한다.

### 예약 가능 수량

기본 계산식:

```text
예약 가능 수량 = 후보 테이블 수 - 유효 예약이 점유한 후보 테이블 수 - 임시 마감 후보 수
```

테이블 조합 사용 시:

```text
예약 가능 조합 수 = 사용 가능한 조합 후보 중 포함 테이블이 모두 비어 있는 조합 수
```

고객에게는 기본적으로 시간대의 예약 가능 여부만 노출한다. 남은 좌석 수 또는 남은 테이블 수 노출은 매장 설정으로 둔다.

### 유효 예약 상태

재고를 점유하는 상태:

- `pending`
- `confirmed`
- `modified`

재고를 복구하거나 미래 재고 점유에서 제외하는 상태:

- `cancelled_by_customer`
- `cancelled_by_restaurant`
- `completed`
- `no_show`

단, `completed`와 `no_show`는 과거 방문 기록으로 보존되며 과거 타임슬롯의 통계에는 반영한다.

### 시간 겹침 판단

두 예약의 시간 범위는 다음 조건이면 겹친 것으로 본다.

```text
existing.starts_at < requested.ends_at
AND existing.ends_at > requested.starts_at
```

종료 시각과 다음 예약 시작 시각이 같은 경우는 겹치지 않는 것으로 본다. 예를 들어 18:00-19:30 예약과 19:30-21:00 예약은 연속 예약으로 허용한다.

### 테이블 배정 우선순위

자동 배정이 필요한 경우 다음 순서를 기본값으로 둔다.

1. 고객 인원을 수용하는 테이블 중 낭비 좌석이 가장 적은 테이블.
2. 같은 조건이면 좌석 유형 우선순위가 높은 테이블.
3. 같은 조건이면 오너가 지정한 정렬 순서가 빠른 테이블.
4. 같은 조건이면 테이블 ID가 작은 테이블.

이 규칙은 예측 가능성을 위한 기본값이며, 오너가 수동 예약 등록 시 특정 테이블을 직접 선택할 수 있다.

## 동시성/락 정책 초안

- 예약 생성, 예약 변경, 취소, 결제 실패 처리는 모두 재고 변경 트랜잭션으로 묶는다.
- 예약 가능 조회는 캐시 또는 읽기 전용 계산을 사용할 수 있으나, 예약 생성 시점에는 반드시 다시 검증한다.
- 예약 생성 트랜잭션에서는 후보 `time_slots` 행과 후보 테이블 점유 범위를 잠근다.
- 관계형 데이터베이스 사용 시 `SELECT ... FOR UPDATE` 또는 동등한 행 잠금을 사용한다.
- 시간 범위 중복은 애플리케이션 검사와 데이터베이스 제약을 함께 둔다.
- PostgreSQL을 사용하는 경우 테이블 ID와 시간 범위에 대한 exclusion constraint를 검토한다.
- 멱등키는 같은 고객 예약 생성 요청의 재시도를 같은 결과로 반환하는 데 사용한다.
- `pending` 예약은 결제 제한 시간 동안만 재고를 점유하며, 만료 작업이 주기적으로 해제한다.
- 락 획득 실패나 직전 선점으로 재고가 사라진 경우 `409 Conflict`를 반환하고 고객에게 다른 시간을 선택하게 한다.

## 유효성 검증과 예외 케이스

### 테이블 설정 검증

- 테이블명은 같은 매장 내 활성 테이블 기준으로 중복될 수 없다.
- 최소 수용 인원은 1 이상이어야 한다.
- 최대 수용 인원은 최소 수용 인원보다 작을 수 없다.
- 예약 이력이 있는 테이블은 삭제할 수 없고 비활성화만 가능하다.
- 비활성 테이블은 신규 예약 후보에서 제외한다.

### 상품/좌석 연결 검증

- 상품에 연결된 테이블은 같은 매장의 활성 테이블이어야 한다.
- 상품 최대 인원이 연결된 모든 테이블의 최대 수용 인원보다 큰 경우 경고를 표시한다.
- 상품 기본 이용 시간은 슬롯 간격보다 짧을 수 없다.
- 상품 예약 가능 시간이 매장 영업시간 밖이면 해당 범위는 슬롯 생성에서 제외한다.

### 예약 생성 검증

- 예약 인원은 매장, 상품, 테이블 수용 인원 제한을 모두 만족해야 한다.
- 예약 시작 시각은 생성된 타임슬롯 또는 정책상 허용된 시작 시각이어야 한다.
- 예약 종료 시각이 영업 종료 또는 브레이크타임을 침범하면 예약할 수 없다.
- 임시 마감된 시간대에는 고객 예약과 오너 수동 예약 모두 기본적으로 차단한다.
- 오너 수동 예약에서 예외 허용이 필요한 경우 별도 권한과 감사 로그가 필요하므로 Phase 5 기본 범위에서는 제외한다.

### 예약 변경/취소 예외

- 변경 대상 시간에 재고가 없으면 기존 예약을 유지한다.
- 취소 정책과 환불 정책은 Phase 4 규칙을 따른다.
- 결제 완료 예약 변경 시 추가 결제 또는 부분 환불이 필요하면 Phase 4 결제/환불 플로우와 연동한다.
- 예약 상태가 `completed` 또는 `no_show`이면 시간 변경을 허용하지 않는다.

### 타임슬롯 생성 예외

- 영업시간이 없거나 매장이 승인 전이면 타임슬롯을 생성하지 않는다.
- 정기 휴무와 임시 휴무 날짜는 생성 대상에서 제외한다.
- 이미 예약이 있는 슬롯은 정책 변경으로 더 이상 예약 불가해져도 자동 삭제하지 않고 운영자 경고로 표시한다.
- 과거 슬롯은 재생성하지 않는다.

## 완료 기준

- 매장 오너가 테이블을 등록, 수정, 비활성화할 수 있다.
- 좌석 유형과 수용 인원 기준으로 예약 가능한 테이블 후보가 계산된다.
- 예약 상품별로 허용 좌석 유형 또는 테이블을 연결할 수 있다.
- 매장 기본 이용 시간과 상품별 이용 시간이 예약 종료 시각 계산에 반영된다.
- 영업시간, 휴무, 상품 정책 기준으로 미래 타임슬롯이 생성된다.
- 고객 예약 가능 시간 조회가 인원, 상품, 좌석, 기존 예약, 임시 마감을 반영한다.
- 예약 생성 시 재고가 트랜잭션 안에서 차감되고 중복 예약이 방지된다.
- 예약 취소, 결제 실패, 결제 대기 만료 시 재고가 복구된다.
- 예약 변경은 새 재고 확보 실패 시 기존 예약을 보존한다.
- 임시 마감 설정과 해제가 고객 예약 가능 시간에 반영된다.
- 동시 예약 요청에서 동일 테이블에 중복 예약이 생성되지 않는다.
- 단체 예약, 대관, POS 연동 기능이 포함되지 않았음을 문서와 화면 범위에서 확인할 수 있다.

## 후속 Phase 연계

- Phase 6 고객 관리/CRM은 예약의 좌석 유형, 방문 이력, 노쇼 이력을 고객 프로필과 연결해 VIP/주의 고객 판단에 활용할 수 있다.
- Phase 7 운영 통계는 타임슬롯별 예약률, 좌석 유형별 점유율, 임시 마감 빈도, 노쇼 영향을 분석할 수 있다.
- 향후 테이블 조합 고도화 단계에서는 수동 조합을 넘어 자동 조합 추천을 검토할 수 있으나, 일반 예약 범위를 넘는 단체 예약과 대관은 별도 Phase로 분리한다.
- 향후 홀 운영 고도화 단계에서는 플로어 맵, 실시간 착석 상태, 직원 권한 분리를 검토할 수 있다.
- POS 연동은 제품 제외 범위에 있으므로 별도 사업 요구가 확정되기 전까지 Phase 5 후속 범위에 포함하지 않는다.
