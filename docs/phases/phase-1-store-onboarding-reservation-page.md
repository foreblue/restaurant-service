# Phase 1. 매장 입점/예약 페이지

## 개요

Phase 1은 식당이 플랫폼에서 예약을 받을 준비를 마치는 단계다. 매장 오너는 매장 기본 정보와 사업자 정보를 등록하고, 플랫폼 관리자의 승인을 받은 뒤 영업시간과 휴무일을 설정해 고유 예약 페이지를 생성한다.

이 Phase의 산출물은 예약 가능한 매장의 기반 데이터와 공개 가능한 예약 페이지다. 고객이 날짜, 시간, 인원, 상품을 선택해 실제 예약을 생성하는 흐름은 Phase 2에서 다룬다.

기준 제품 방향에 따라 플랫폼 내부 맛집 탐색, 웨이팅, 지도, 통합 검색은 제공하지 않는다. 고객은 매장이 공유한 직접 링크, SNS, QR, 홈페이지 위젯, 네이버 플레이스 등의 외부 진입점으로 예약 페이지에 접근한다.

## 목표와 비목표

### 목표

- 매장 오너가 단일 매장을 등록하고 작성중 상태로 저장할 수 있다.
- 매장 오너가 사업자등록번호, 사업자등록증, 대표자/담당자 정보를 제출할 수 있다.
- 플랫폼 관리자가 매장 입점 신청을 검토하고 승인 또는 반려할 수 있다.
- 승인된 매장 오너가 요일별 영업시간, 브레이크타임, 정기 휴무, 임시 휴무를 설정할 수 있다.
- 승인된 매장에 고유 예약 페이지 URL을 생성하고 공개/비공개 상태를 관리할 수 있다.
- 모든 승인, 반려, 공개 상태 변경, 주요 설정 변경을 운영 로그로 남긴다.

### 비목표

- 고객 예약 생성, 예약 변경, 예약 취소는 Phase 2 범위다.
- 예약 상품, 예약금, 결제, 환불, 카드 보증은 Phase 2와 Phase 4 범위다.
- 테이블, 좌석 유형, 타임슬롯 재고, 중복 예약 방지는 Phase 5 범위다.
- 고객 관리, CRM, 방문/노쇼 이력 관리는 Phase 6 범위다.
- 예약 통계와 운영 리포트는 Phase 7 범위다.
- 웨이팅, 지도 기반 탐색, 통합 검색은 제품 범위에서 제외한다.
- POS 연동, 멀티지점 관리, 직원 권한 분리, 정산 자동화는 MVP에서 제외한다.

## 사용자 역할과 권한 범위

MVP에서는 직원 권한을 제공하지 않는다. 매장 계정은 오너 1명이 단일 매장을 직접 운영하는 구조로 본다.

| 역할 | 권한 범위 |
| --- | --- |
| 고객 | 공개된 예약 페이지 조회. Phase 1에서는 실제 예약 생성 불가 |
| 매장 오너 | 본인 매장 등록, 사업자 정보 제출, 승인 요청, 영업시간/휴무 설정, 예약 페이지 공개 상태 관리 |
| 플랫폼 관리자 | 전체 매장 입점 신청 조회, 승인/반려, 매장 운영 정지/해제, 운영 로그 조회 |

권한 정책:

- 매장 오너는 `owner_user_id`가 본인인 매장만 조회/수정할 수 있다.
- 매장 오너는 승인 결과, 관리자 검토 메모, 타 매장의 신청 정보를 수정하거나 볼 수 없다.
- 플랫폼 관리자는 승인 상태와 운영 상태를 변경할 수 있으나, 오너가 제출한 사업자 원본 정보는 직접 수정하지 않는다.
- MVP에서는 `restaurant_users`, `staff_roles`, 지점별 권한, 직원 초대 기능을 만들지 않는다.

## 주요 사용자 시나리오

1. 매장 오너가 매장 기본 정보를 작성중으로 저장한다.
   - 오너는 매장명, 전화번호, 주소, 업종, 소개 문구, 대표 이미지를 입력한다.
   - 필수값이 부족해도 작성중 저장은 가능하나 승인 요청은 불가하다.

2. 매장 오너가 사업자/소유자 인증 정보를 제출한다.
   - 오너는 사업자등록번호, 대표자명, 담당자 정보, 사업자등록증 파일을 입력한다.
   - 담당자 휴대폰 인증을 완료한 뒤 승인 요청을 보낸다.

3. 플랫폼 관리자가 입점 신청을 승인한다.
   - 관리자는 제출 정보를 확인하고 승인 처리한다.
   - 매장 상태는 승인 완료가 되고, 예약 페이지 URL이 생성된다.
   - 오너에게 승인 알림이 발송된다.

4. 플랫폼 관리자가 입점 신청을 반려한다.
   - 관리자는 반려 사유를 필수로 입력한다.
   - 매장 상태는 반려가 되고, 오너는 사유를 확인한 뒤 정보를 수정해 다시 승인 요청할 수 있다.

5. 매장 오너가 영업시간과 휴무일을 설정한다.
   - 요일별 영업 구간과 브레이크타임을 입력한다.
   - 정기 휴무와 특정 날짜 임시 휴무를 등록한다.
   - 유효성 검증을 통과하면 예약 페이지에 영업 정보가 반영된다.

6. 매장 오너가 예약 페이지를 공개한다.
   - 승인 완료, 필수 매장 정보, 영업시간 설정이 모두 충족되면 공개 가능하다.
   - 공개된 URL은 외부 채널에 공유할 수 있다.
   - Phase 1에서는 고객 예약 버튼을 노출하지 않거나, 예약 준비중 상태로 표시한다.

## 기능 상세

### 매장등록

매장등록은 예약 운영의 기준이 되는 단일 매장을 생성하는 기능이다.

입력 필드:

| 필드 | 필수 | 설명 |
| --- | --- | --- |
| 매장명 | 예 | 고객에게 노출되는 이름 |
| 대표 이미지 | 아니오 | 예약 페이지 상단 이미지. JPG, PNG, WebP 허용 |
| 소개 문구 | 아니오 | 예약 페이지에 노출되는 짧은 설명 |
| 전화번호 | 예 | 고객 안내와 관리자 확인에 사용 |
| 주소 | 예 | 방문 안내와 사업자 검증에 사용. 지도 기능은 제공하지 않음 |
| 상세 주소 | 아니오 | 층, 호수 등 |
| 업종/음식 종류 | 예 | 한식, 일식, 오마카세 등 단일 또는 다중 선택 가능 |
| 예약 페이지 경로 후보 | 아니오 | 영문 소문자, 숫자, 하이픈 기반 slug |

처리 규칙:

- 계정당 매장 1개만 등록할 수 있다.
- 최초 생성 시 매장 상태는 `draft`다.
- 작성중 상태에서는 오너가 기본 정보를 자유롭게 수정할 수 있다.
- 승인 요청 이후에는 검토에 영향을 주는 필드 변경을 제한한다. 수정이 필요하면 승인 요청을 철회하거나 반려 후 재제출한다.
- 주소는 필수지만 좌표 변환, 주변 검색, 지도 노출은 하지 않는다.
- 예약 페이지 경로 후보가 비어 있으면 승인 시 시스템이 매장명 기반 slug를 생성한다.

### 사업자/소유자 인증

사업자/소유자 인증은 가짜 매장 생성, 무단 입점, 예약금 사기 위험을 줄이기 위한 관리자 승인 전 단계다.

입력 필드:

| 필드 | 필수 | 설명 |
| --- | --- | --- |
| 사업자등록번호 | 예 | 숫자 10자리 |
| 상호명 | 예 | 사업자등록증 기준 상호 |
| 대표자명 | 예 | 사업자등록증 기준 대표자 |
| 사업장 주소 | 예 | 사업자등록증 또는 실제 매장 주소 |
| 사업자등록증 파일 | 예 | PDF, JPG, PNG 허용 |
| 담당자 이름 | 예 | 플랫폼 운영 연락 담당자 |
| 담당자 휴대폰 번호 | 예 | 인증 코드 발송 대상 |
| 담당자 이메일 | 아니오 | 승인 결과 알림 보조 채널 |

처리 규칙:

- 사업자등록번호는 숫자만 저장하고 화면 표시 시 `000-00-00000` 형식으로 표시한다.
- 동일 사업자등록번호로 승인 완료된 매장이 있으면 신규 승인 요청을 막는다.
- 담당자 휴대폰 인증은 승인 요청 전에 완료되어야 한다.
- 사업자등록증 파일은 비공개 파일로 저장하고 오너 본인과 플랫폼 관리자만 접근할 수 있다.
- 승인 완료 후 사업자등록번호, 대표자명, 사업자등록증 파일을 변경하려면 재승인 요청이 필요하다.

### 관리자 승인/반려

관리자는 입점 신청 목록에서 제출된 매장을 검토한다.

승인 처리:

- 신청 상태가 `submitted`인 건만 승인할 수 있다.
- 승인 시 매장 상태를 `approved`로 변경한다.
- 승인 시 예약 페이지 slug를 확정하고, 아직 페이지 레코드가 없으면 생성한다.
- 승인 일시, 승인 관리자, 검토 메모를 저장한다.
- 오너에게 승인 알림을 발송한다.

반려 처리:

- 반려 사유는 필수다.
- 반려 시 매장 상태를 `rejected`로 변경한다.
- 오너는 반려 사유를 확인하고 정보를 수정해 새 승인 요청을 제출할 수 있다.
- 기존 반려 신청 이력은 보존한다.

운영 정지:

- 승인 완료 후 사기 의심, 법적 이슈, 운영 정책 위반이 있으면 관리자가 `suspended`로 변경할 수 있다.
- 운영 정지 상태의 예약 페이지는 공개 URL 접근 시 운영 정지 안내 또는 404 정책 중 하나로 처리한다. MVP 기본값은 404다.

### 영업시간/휴무 설정

영업시간은 실제 예약 가능 시간 계산의 기반 데이터다. Phase 1에서는 예약 가능 슬롯을 만들지 않고, Phase 2와 Phase 5에서 이 설정을 사용한다.

요일별 영업시간:

- 요일은 월요일부터 일요일까지 7개로 관리한다.
- 하루에 여러 영업 구간을 등록할 수 있다.
- 브레이크타임은 영업 구간 사이의 비영업 시간으로 표현한다.
- 자정을 넘는 영업시간은 MVP에서 한 구간으로 저장하지 않고, 전날과 다음날 구간으로 나누어 등록한다.

예시:

| 요일 | 영업 구간 |
| --- | --- |
| 월 | 11:30-15:00, 17:30-22:00 |
| 화 | 휴무 |
| 토 | 12:00-22:00 |

휴무 설정:

- 정기 휴무: 매주 특정 요일 휴무
- 월간 휴무: 매월 특정 일자 또는 특정 주차/요일 휴무
- 임시 휴무: 특정 날짜 전체 휴무
- 임시 단축영업: 특정 날짜의 일부 시간만 휴무

처리 규칙:

- 시간대 기준은 `Asia/Seoul`을 기본값으로 한다.
- 영업 구간은 시작 시간이 종료 시간보다 빨라야 한다.
- 같은 요일의 영업 구간은 서로 겹칠 수 없다.
- 임시 휴무는 정기 영업시간보다 우선한다.
- 승인 전에도 영업시간을 작성할 수 있으나, 공개 페이지 반영은 승인 후 가능하다.

### 예약 페이지 생성/공개 상태

예약 페이지는 매장이 외부 채널에 공유하는 고객 진입점이다.

URL 정책:

```text
https://service.example.com/r/{slug}
```

처리 규칙:

- 매장 승인 시 `slug`를 확정하고 예약 페이지를 생성한다.
- `slug`는 전체 서비스에서 유일해야 한다.
- 허용 문자는 영문 소문자, 숫자, 하이픈이다.
- 예약어(`admin`, `api`, `login`, `owner`, `reserve`, `static` 등)는 사용할 수 없다.
- 승인 전에는 오너 로그인 상태에서만 미리보기를 제공할 수 있다.

공개 조건:

- 매장 상태가 `approved`여야 한다.
- 예약 페이지 상태가 `public`이어야 한다.
- 매장명, 전화번호, 주소, 업종, 사업자 승인 정보가 존재해야 한다.
- 최소 1개 이상의 영업 구간 또는 명시적 전체 휴무 설정이 있어야 한다.

페이지 노출 정보:

- 매장명
- 대표 이미지
- 소개 문구
- 전화번호
- 주소와 상세 주소
- 업종/음식 종류
- 요일별 영업시간
- 정기/임시 휴무 안내
- 예약 준비중 안내 또는 Phase 2 예약 플로우 진입점

Phase 1에서는 예약 상품 선택, 날짜/시간 선택, 인원 선택, 고객 정보 입력, 결제 버튼을 구현하지 않는다.

| 공개 | 예약 페이지 | 공개된 매장 정보와 영업시간 표시 |
| 공개 | 비공개/미승인 접근 | 비공개, 미승인, 정지 상태 접근 처리 |

관리자 목록의 검색은 입점 신청 운영을 위한 관리자 내부 필터다. 고객 대상 통합 맛집 검색은 제공하지 않는다.

## 상태/정책

### 매장 상태

| 상태 | 설명 | 주요 전환 |
| --- | --- | --- |
| `draft` | 오너가 작성중 | 생성 시 기본값 |
| `approval_requested` | 승인 요청됨 | 오너가 신청 제출 |
| `approved` | 관리자 승인 완료 | 관리자가 승인 |
| `rejected` | 관리자 반려 | 관리자가 반려 |
| `suspended` | 운영 정지 | 관리자가 정지 |

전환 정책:

- `draft` -> `approval_requested`: 필수 정보와 연락처 인증 완료 필요
- `approval_requested` -> `approved`: 관리자 승인
- `approval_requested` -> `rejected`: 관리자 반려
- `rejected` -> `approval_requested`: 오너 수정 후 재제출
- `approved` -> `suspended`: 관리자 운영 정지
- `suspended` -> `approved`: 관리자 정지 해제. 단, 예약 페이지는 자동 공개하지 않는다.

### 승인 상태

| 상태 | 설명 |
| --- | --- |
| `draft` | 사업자 인증 정보 작성중 |
| `submitted` | 관리자 검토 대기 |
| `approved` | 승인 완료 |
| `rejected` | 반려 완료 |
| `cancelled` | 오너가 제출 전 또는 검토 전 철회 |

정책:

- 매장당 동시에 검토중인 신청은 1개만 허용한다.
- 반려 사유는 오너에게 노출되므로 구체적으로 작성한다.
- 승인/반려 이후 신청 원본은 수정하지 않고 새 신청으로 재제출한다.

### 페이지 공개 상태

| 상태 | 설명 |
| --- | --- |
| `draft` | 생성 전 또는 초기 작성중 |
| `private` | URL은 존재하지만 공개하지 않음 |
| `public` | 고객이 직접 URL로 접근 가능 |
| `disabled` | 운영 정지, 정책 위반 등으로 비활성 |

정책:

- 승인 완료 전에는 `public`이 될 수 없다.
- 오너가 비공개로 전환하면 공개 URL 접근은 404를 반환한다.
- 관리자가 운영 정지하면 페이지 상태를 `disabled`로 바꾸고 오너가 직접 공개할 수 없게 한다.
- 정지 해제 후에는 `private`로 복구하고 오너가 다시 공개해야 한다.

## 유효성 검증과 예외 케이스

### 매장 기본 정보

- 매장명은 2자 이상 80자 이하로 제한한다.
- 전화번호는 국내 전화번호 형식을 우선 지원한다.
- 주소는 기본 주소가 필수이며, 지도 좌표는 요구하지 않는다.
- 대표 이미지는 파일 크기 10MB 이하, JPG/PNG/WebP만 허용한다.
- 업종/음식 종류는 최소 1개 이상 선택해야 승인 요청이 가능하다.

### 사업자/소유자 인증

- 사업자등록번호는 숫자 10자리만 허용한다.
- 이미 승인 완료된 사업자등록번호는 중복 승인할 수 없다.
- 사업자등록증 파일은 10MB 이하, PDF/JPG/PNG만 허용한다.
- 담당자 휴대폰 인증 코드는 5분 동안 유효하다.
- 인증 실패가 5회 누적되면 10분 동안 재시도를 제한한다.
- 승인 요청 시 필수 파일이 삭제되었거나 접근 불가능하면 제출을 막는다.

### 승인/반려

- 이미 처리된 신청은 다시 승인 또는 반려할 수 없다.
- 반려 시 반려 사유가 없으면 400 오류를 반환한다.
- 관리자가 검토 중 오너가 신청을 철회하면 관리자 처리 요청은 409 오류를 반환한다.
- 승인 처리 중 slug 중복이 발생하면 시스템이 숫자 suffix를 붙여 재시도한다.

### 영업시간/휴무

- `opens_at`은 `closes_at`보다 빨라야 한다.
- 같은 요일의 영업 구간은 겹칠 수 없다.
- 하루 전체 휴무(`is_closed=true`)와 영업 구간은 동시에 저장할 수 없다.
- 자정을 넘는 영업은 다음날 구간으로 나눠 입력하도록 안내한다.
- 임시 휴무 날짜가 과거이면 저장은 허용하되 예약 가능 계산에는 사용하지 않는다.
- 임시 휴무와 영업시간이 충돌하면 임시 휴무가 우선한다.

### 예약 페이지

- slug는 3자 이상 60자 이하로 제한한다.
- slug는 `^[a-z0-9]+(?:-[a-z0-9]+)*$` 형식만 허용한다.
- 예약어 slug는 사용할 수 없다.
- 공개 조건을 만족하지 못하면 공개 API는 `publish_blocked_reasons`를 반환한다.
- 공개 페이지에서 비공개 정보가 응답에 포함되면 보안 결함으로 본다.

## 알림/운영 로그

### 알림

필수 알림:

- 오너에게 승인 요청 접수 알림
- 관리자에게 신규 입점 신청 알림
- 오너에게 승인 완료 알림
- 오너에게 반려 알림과 반려 사유 안내
- 오너에게 예약 페이지 공개/비공개 변경 알림
- 오너에게 운영 정지/해제 알림

채널 우선순위:

- 서비스 내부 알림은 기본으로 생성한다.
- 승인/반려/정지처럼 중요한 이벤트는 이메일 또는 SMS를 함께 발송한다.
- 카카오 알림톡은 공통 알림 인프라가 준비된 경우 사용한다.

### 운영 로그

로그 대상:

- 매장 생성과 기본 정보 변경
- 사업자 인증 정보 저장과 승인 요청
- 관리자 승인, 반려, 운영 정지, 정지 해제
- 영업시간 생성/수정/삭제
- 휴무 규칙 생성/수정/삭제
- 예약 페이지 slug 변경, 공개, 비공개, 비활성화

로그 정책:

- 관리자 행위는 `actor_user_id`, IP, User-Agent를 반드시 저장한다.
- 사업자등록번호와 첨부파일 원문은 로그에 저장하지 않는다.
- 변경 전/후 값은 민감정보를 마스킹한 형태로 저장한다.

## 완료 기준

- 오너가 매장 기본 정보를 작성중으로 생성하고 다시 조회할 수 있다.
- 오너가 필수 사업자/소유자 인증 정보를 입력하고 승인 요청을 제출할 수 있다.
- 필수값, 파일 형식, 연락처 인증, 사업자등록번호 중복 검증이 동작한다.
- 관리자가 제출된 입점 신청 목록과 상세를 확인할 수 있다.
- 관리자가 승인 또는 반려를 처리할 수 있고, 반려 사유가 오너에게 표시된다.
- 승인 시 매장 상태가 승인 완료로 바뀌고 예약 페이지가 생성된다.
- 오너가 요일별 영업시간, 브레이크타임, 정기 휴무, 임시 휴무를 저장할 수 있다.
- 오너가 공개 조건을 만족한 예약 페이지를 공개/비공개로 전환할 수 있다.
- 공개 예약 페이지는 직접 URL로 접근 가능하며 매장 정보와 영업시간을 표시한다.
- 비공개, 미승인, 운영 정지 페이지는 공개 API에서 404로 처리된다.
- 주요 변경에 대해 알림과 운영 로그가 생성된다.
- 직원 권한, POS, 멀티지점, 정산 자동화, 웨이팅, 지도, 통합 검색 기능이 포함되지 않는다.

## 후속 Phase 연계

- Phase 2 기본 예약
  - Phase 1의 공개 예약 페이지에 예약 상품 선택, 날짜/시간/인원 선택, 고객 정보 입력, 예약 확정/취소 흐름을 연결한다.
  - `business_hours`, `holiday_rules`, `reservation_pages`를 예약 가능 일자 계산의 입력으로 사용한다.

- Phase 3 오너 예약 관리
  - 승인된 매장과 공개 예약 페이지를 기준으로 예약 리스트, 캘린더, 예약 상세, 수동 예약 등록 기능을 제공한다.

- Phase 4 예약금/환불
  - 승인된 사업자 정보와 매장 정보를 결제/환불 기록, 수동 정산 확인의 기준 데이터로 사용한다.
  - MVP에서는 정산 자동화를 만들지 않는다.

- Phase 5 좌석/재고 관리
  - Phase 1의 영업시간과 휴무 설정을 기반으로 테이블, 좌석 유형, 타임슬롯 재고를 계산한다.

- Phase 6 고객 관리/CRM
  - 공개 예약 페이지로 유입된 고객 예약 이력을 고객 프로필, 요청사항, 알레르기, VIP/주의 고객 표시와 연결한다.

- Phase 7 운영 통계
  - 매장 승인일, 페이지 공개일, 영업시간 설정을 기준으로 예약 수, 취소 수, 노쇼 수, 예약금 매출 등 통계를 집계한다.
