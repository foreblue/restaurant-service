# 식당 예약 플랫폼 기획 정리

## 1. 제품 방향

식당 예약 플랫폼은 맛집 탐색 서비스가 아니라, 식당이 자체 예약 링크를 통해 예약을 받고 운영할 수 있는 예약 관리 SaaS로 설계한다.

고객은 식당이 공유한 예약 페이지, 홈페이지 위젯, SNS 링크, QR, 네이버 플레이스 또는 인스타그램 프로필 링크 등을 통해 예약한다. 플랫폼 내부에서 주변 맛집 찾기, 지도 기반 탐색, 통합 검색, 웨이팅은 제공하지 않는다.

## 2. 참고 서비스 조합

MVP와 이후 확장 설계는 다음 서비스를 조합해 참고한다.

| 서비스 | 참고 포인트 |
| --- | --- |
| 캐치테이블 | 예약 상품, 예약금, 노쇼 방지, 매장용 예약 운영 |
| 네이버 예약 | 예약 상품, 사업자용 예약 관리, 예약 페이지, 결제 연동 구조 |
| OpenTable | 예약 관리, 테이블 운영, 매장용 캘린더와 예약 흐름 |
| Resy | 고객 방문 이력, 고객 취향, 기념일, 게스트 인텔리전스 |
| Tock | 선결제 예약, 코스 예약, 노쇼 방지, 예약 상품화 |
| SevenRooms | CRM, 고객 세그먼트, 알림, 마케팅 확장 구조 |

## 3. 제외 범위

제품 전체에서 우선 제외하는 기능은 다음과 같다.

| 제외 기능 | 이유 |
| --- | --- |
| 웨이팅 | 예약 관리 서비스에 집중하기 위함 |
| 지도 | 위치 기반 탐색 서비스를 만들지 않음 |
| 검색 | 통합 맛집 검색/발견 플랫폼을 만들지 않음 |

MVP 단계에서 추가로 제외하는 기능은 다음과 같다.

| MVP 제외 기능 | MVP 처리 방식 |
| --- | --- |
| 직원 권한 | 오너/관리자 계정 1개만 제공 |
| 단체 예약 | 일반 예약만 제공, 예: 1-8인 제한 |
| POS 연동 | 방문 완료, 노쇼, 결제 확인은 수동 처리 |
| 멀티지점 관리 | 계정당 매장 1개 기준 |
| 정산 자동화 | 결제/환불 내역 조회만 제공하고 정산은 수동 처리 |

## 4. 사용자 역할

| 역할 | 설명 |
| --- | --- |
| 고객 | 예약 생성, 변경, 취소, 예약금 결제, 예약 내역 확인 |
| 매장 오너 | 매장 정보 설정, 예약 상품 관리, 예약 운영, 고객 메모 관리 |
| 플랫폼 관리자 | 매장 승인, 결제/환불 모니터링, 분쟁 대응, 운영 관리 |

MVP에서는 매장 직원과 권한 분리는 제공하지 않는다.

## 5. 매장등록

매장등록은 식당을 예약 가능한 운영 단위로 플랫폼에 입점시키는 기능이다. 단순히 식당 이름을 입력하는 것이 아니라, 예약 페이지, 영업시간, 예약 정책, 상품, 결제/정산 정보와 연결되는 초기 세팅이다.

### 5.1 등록 정보

- 매장명
- 대표 이미지
- 소개 문구
- 전화번호
- 주소
- 업종/음식 종류
- 사업자등록번호
- 대표자명
- 담당자 이름
- 담당자 연락처
- 매장 상태: 작성중, 승인 요청, 승인 완료, 반려, 운영 정지

지도와 검색은 제외하지만 주소는 예약 확인, 고객 방문 안내, 사업자 검증, 결제/환불 기록에 필요하다.

### 5.2 사업자/소유자 인증

- 사업자등록증 업로드
- 대표자 또는 담당자 정보 입력
- 연락처 인증
- 관리자 승인 요청
- 승인/반려 처리

이 절차는 가짜 매장 생성, 예약금 사기, 무단 입점 문제를 방지하기 위해 필요하다.

### 5.3 예약 페이지 생성

매장 승인 후 고유 예약 페이지를 생성한다.

예시:

```text
https://service.example.com/r/restaurant-name
https://service.example.com/reserve/12345
```

식당은 이 링크를 홈페이지, SNS, QR, 문자, 네이버 플레이스 등에 연결해 고객 예약을 받을 수 있다.

## 6. MVP 기능 범위

MVP는 단일 매장 오너가 직접 운영하는 예약 관리 서비스로 정의한다.

### 6.1 매장 설정

- 매장 기본 정보 등록
- 사업자 정보 등록
- 관리자 승인
- 예약 페이지 생성
- 요일별 영업시간
- 브레이크타임
- 정기 휴무
- 임시 휴무
- 예약 오픈 기간
- 예약 마감 시간
- 최소/최대 예약 인원
- 기본 이용 시간
- 30분/60분 단위 타임슬롯

### 6.2 예약 상품

예약 상품은 고객이 선택할 수 있는 예약 유형이다.

예시:

- 일반 좌석 예약
- 런치 코스
- 디너 코스
- 오마카세
- 룸 예약

상품별 설정:

- 상품명
- 설명
- 가격
- 예약금
- 선결제 여부
- 최소/최대 인원
- 예약 가능 요일
- 예약 가능 시간
- 취소 정책
- 재고 수량
- 노출 여부

단체 예약, 대관, 별도 견적, 상담 기반 예약은 MVP에서 제외한다.

### 6.3 고객 예약

고객 예약 플로우:

1. 식당 예약 페이지 진입
2. 예약 상품 선택
3. 날짜 선택
4. 시간 선택
5. 인원 선택
6. 고객 정보 입력
7. 요청사항 입력
8. 예약금 결제 또는 카드 보증
9. 예약 확정
10. 알림 발송

고객 입력 정보:

- 이름
- 휴대폰 번호
- 이메일 선택
- 방문 인원
- 날짜/시간
- 요청사항
- 알레르기/기념일 선택
- 마케팅 수신 동의 선택

예약 상태:

| 상태 | 설명 |
| --- | --- |
| pending | 예약 대기 |
| confirmed | 예약 확정 |
| modified | 예약 변경됨 |
| cancelled_by_customer | 고객 취소 |
| cancelled_by_restaurant | 매장 취소 |
| completed | 방문 완료 |
| no_show | 노쇼 |

### 6.4 매장 예약 관리

- 일별 예약 리스트
- 캘린더 보기
- 예약 상세 확인
- 수동 예약 등록
- 전화 예약 등록
- 예약 시간 변경
- 인원 변경
- 예약 취소
- 방문 완료 처리
- 노쇼 처리
- 고객 요청사항 메모

### 6.5 결제/환불

지원 결제 방식:

- 예약금 결제
- 전액 선결제
- 카드 보증
- 현장 결제
- 무료 예약

취소 정책 예시:

- 방문 48시간 전 무료 취소
- 방문 24시간 전 예약금 일부 환불
- 당일 취소 환불 불가
- 노쇼 시 예약금 미환불
- 매장 취소 시 전액 환불

MVP에서는 정산 자동화를 제공하지 않고, 플랫폼 관리자가 결제/환불 내역을 조회해 수동 정산한다.

### 6.6 알림

필수 알림:

- 예약 확정
- 예약 변경
- 예약 취소
- 결제 완료
- 환불 완료
- 방문 전 리마인드
- 노쇼 처리 안내

채널:

- SMS
- 카카오 알림톡
- 이메일

권장 기본값:

- 예약 직후 확정 알림
- 방문 24시간 전 리마인드
- 방문 2시간 전 리마인드
- 취소 마감 전 안내

### 6.7 고객 관리

- 고객 이름
- 전화번호
- 예약 이력
- 방문 이력
- 노쇼 이력
- 요청사항
- 알레르기
- 기념일
- 매장 메모
- VIP 표시
- 주의 고객 표시

### 6.8 기본 통계

- 예약 수
- 방문 완료 수
- 취소 수
- 노쇼 수
- 예약금 매출
- 환불 금액
- 시간대별 예약률

## 7. 모듈별 기능 구성

시스템은 사용자 FE, 사업자 FE, 백엔드의 3개 모듈로 구성한다. 사용자 FE는 고객 예약 경험, 사업자 FE는 매장 운영 경험, 백엔드는 예약 도메인 규칙과 외부 연동을 담당한다.

### 7.1 사용자 FE

사용자 FE는 고객이 식당에서 공유한 예약 링크에 진입해 예약을 완료하고, 이후 예약 상태를 확인하거나 취소하는 화면을 담당한다.

| 기능 단위 | 포함 기능 |
| --- | --- |
| 예약 페이지 | 매장 소개, 대표 이미지, 주소/전화번호, 예약 가능 상품 노출, 예약 정책 안내 |
| 예약 생성 | 상품 선택, 날짜 선택, 시간 선택, 인원 선택, 고객 정보 입력, 요청사항 입력 |
| 예약금/보증 결제 | 예약금 결제, 전액 선결제, 카드 보증, 결제 결과 확인 |
| 예약 완료 | 예약 확정 화면, 예약 번호/상태 표시, 방문 정보 안내 |
| 예약 조회 | 휴대폰 인증 기반 예약 내역 조회, 예약 상세 확인 |
| 예약 변경/취소 | 취소 가능 여부 확인, 취소 정책 표시, 예약 취소 요청, 환불 예상 정보 표시 |
| 고객 동의 | 개인정보 수집 동의, 마케팅 수신 동의 선택, 알레르기/기념일 정보 입력 |

사용자 FE에는 맛집 탐색, 지도 검색, 통합 검색, 웨이팅 화면을 포함하지 않는다.

### 7.2 사업자 FE

사업자 FE는 매장 오너가 입점 신청부터 예약 운영까지 수행하는 관리 화면을 담당한다. MVP에서는 계정당 매장 1개와 오너/관리자 계정 1개를 기준으로 한다.

| 기능 단위 | 포함 기능 |
| --- | --- |
| 계정/로그인 | 사업자 회원가입, 로그인, 비밀번호 재설정, 기본 계정 정보 관리 |
| 매장 입점 | 매장 기본 정보 입력, 사업자 정보 입력, 사업자등록증 업로드, 승인 요청, 승인 상태 확인 |
| 매장 설정 | 영업시간, 브레이크타임, 정기 휴무, 임시 휴무, 예약 오픈 기간, 예약 마감 시간 설정 |
| 예약 페이지 관리 | 예약 페이지 미리보기, 공개 상태 관리, 공유 링크/QR 확인 |
| 예약 상품 관리 | 예약 상품 생성/수정/삭제, 가격, 예약금, 선결제 여부, 예약 가능 요일/시간, 노출 여부 설정 |
| 예약 운영 | 일별 예약 리스트, 캘린더, 예약 상세, 수동 예약 등록, 전화 예약 등록 |
| 예약 상태 관리 | 예약 변경, 매장 취소, 방문 완료 처리, 노쇼 처리 |
| 고객 관리 | 고객 프로필, 예약/방문/노쇼 이력, 요청사항, 알레르기, 기념일, 매장 메모, VIP/주의 고객 표시 |
| 결제/환불 확인 | 결제 내역, 환불 내역, 예약금 상태, 수동 환불 요청 또는 처리 결과 확인 |
| 운영 통계 | 예약 수, 방문 완료 수, 취소 수, 노쇼 수, 예약금 매출, 환불 금액, 시간대별 예약률 |

직원 권한 분리, 멀티지점 관리, POS 연동, 정산 자동화는 MVP 사업자 FE 범위에 포함하지 않는다.

### 7.3 백엔드

백엔드는 사용자 FE와 사업자 FE가 사용하는 API, 예약 가능 여부 계산, 결제/환불, 알림, 관리자 운영 기능을 담당한다.

| 기능 단위 | 포함 기능 |
| --- | --- |
| 인증/계정 | 고객 예약 조회 인증, 사업자 로그인, 세션/토큰 관리, 권한 검증 |
| 매장 도메인 | 매장 정보, 사업자 정보, 입점 신청, 승인/반려, 운영 상태 관리 |
| 예약 페이지 API | 공개 예약 페이지 조회, 예약 상품 조회, 예약 가능 날짜/시간 조회 |
| 예약 엔진 | 예약 생성, 예약 변경, 예약 취소, 예약 상태 전이, 중복 예약 방지 |
| 영업시간/재고 계산 | 영업시간, 브레이크타임, 휴무, 예약 오픈/마감 시간, 상품별 재고, 타임슬롯 계산 |
| 예약 상품 도메인 | 상품 가격, 예약금, 선결제 여부, 예약 가능 요일/시간, 취소 정책 관리 |
| 결제/환불 | PG 연동, 예약금 결제, 전액 선결제, 카드 보증, 환불 요청, 환불 결과 저장 |
| 알림 | SMS, 카카오 알림톡, 이메일 발송, 예약 확정/변경/취소/리마인드/환불 알림 |
| 고객/CRM | 고객 프로필, 예약 이력, 방문 이력, 노쇼 이력, 매장 메모, VIP/주의 고객 플래그 |
| 사업자 운영 API | 예약 리스트, 캘린더, 예약 상세, 수동 예약 등록, 방문 완료, 노쇼 처리 |
| 플랫폼 운영 기능 | 매장 승인/반려, 결제/환불 모니터링, 분쟁 대응용 조회, 운영 정지 처리 |
| 통계 집계 | 예약 수, 방문 완료, 취소, 노쇼, 예약금 매출, 환불 금액, 시간대별 예약률 집계 |
| 감사 로그 | 예약 변경 이력, 결제/환불 이력, 관리자 승인 이력, 사업자 주요 설정 변경 이력 |

플랫폼 관리자 전용 화면은 별도 모듈로 두지 않는다. MVP에서는 백엔드 운영 API 또는 내부 운영 도구로 처리하고, 운영 규모가 커지면 관리자 FE를 별도 모듈로 분리한다.

## 9. Phase 계획

각 Phase는 배포 가능한 완결 기능 단위로 나눈다.

| Phase | 목표 | 완결 기능 |
| --- | --- | --- |
| Phase 1. 매장 입점/예약 페이지 | 식당이 예약을 받을 준비를 완료 | 매장등록, 사업자 정보, 관리자 승인, 영업시간 설정, 예약 페이지 생성 |
| Phase 2. 기본 예약 | 고객이 실제 예약 가능 | 날짜/시간/인원 선택, 예약 상품 선택, 고객 정보 입력, 예약 확정/취소, 예약 알림 |
| Phase 3. 오너 예약 관리 | 매장이 예약을 운영 가능 | 예약 리스트, 캘린더, 예약 상세, 수동 예약 등록, 예약 변경/취소, 방문 완료/노쇼 처리 |
| Phase 4. 예약금/환불 | 노쇼 방지와 결제 흐름 완성 | 예약금 결제, 전액 선결제, 취소 정책, 환불 처리, 결제/환불 알림 |
| Phase 5. 좌석/재고 관리 | 시간대별 수용량을 정교하게 통제 | 테이블 등록, 좌석 유형, 이용 시간, 타임슬롯 재고, 중복 예약 방지 |
| Phase 6. 고객 관리/CRM | 재방문과 리스크 관리 가능 | 고객 프로필, 예약/방문/노쇼 이력, 요청사항, 알레르기, VIP/주의 고객 표시 |
| Phase 7. 운영 통계 | 매장이 예약 성과를 파악 | 예약 수, 방문 완료, 취소, 노쇼, 예약금 매출, 환불 금액, 시간대별 예약률 |

MVP의 최소 완성 지점은 Phase 1-4로 본다. Phase 5-7은 예약 운영 품질과 매장 관리 효율을 높이는 확장 단계다.

## 10. MVP 한 줄 정의

MVP는 매장 1곳이 예약 페이지를 만들고, 예약 상품과 영업시간을 설정하고, 고객 예약을 받고, 예약금 기반으로 노쇼를 방지하며, 오너가 직접 예약을 관리하는 서비스다.
