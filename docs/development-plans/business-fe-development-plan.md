# 사업자 FE 개발 계획

## 1. 개요

사업자 FE는 매장 오너가 입점 신청, 매장 설정, 예약 상품 관리, 예약 운영, 결제/환불 확인, 고객 관리, 운영 통계를 수행하는 운영 SPA다.

기술 기준은 `apps/business-web`의 Vite React, TypeScript strict mode, Tailwind CSS, shadcn/ui 기반 관리 UI, TanStack Query, Zustand, TanStack Table, React Hook Form, Zod, OpenAPI 기반 API client를 따른다.

MVP는 계정당 매장 1개, 오너/관리자 계정 1개를 기준으로 한다. 직원 권한, 멀티지점, POS 연동, 정산 자동화는 제외한다.

## 2. 개발 목표

- 매장 오너가 입점 신청부터 예약 운영까지 한 화면 체계에서 수행할 수 있게 한다.
- 예약 운영 화면은 반복 사용을 고려해 리스트, 캘린더, 상세 패널 중심으로 빠르게 조작 가능하게 만든다.
- 예약 상품, 영업시간, 취소/결제 정책 등 설정 화면은 검증과 미리보기를 제공한다.
- 결제/환불은 정산 자동화가 아니라 상태 확인과 수동 운영 보조에 집중한다.

## 3. 모듈 책임

| 영역 | 책임 |
| --- | --- |
| 계정/로그인 | 사업자 로그인, 세션 유지, 비밀번호 재설정 |
| 매장 입점 | 매장 기본 정보, 사업자 정보, 서류 업로드, 승인 상태 확인 |
| 매장 설정 | 영업시간, 브레이크타임, 휴무, 예약 오픈/마감 설정 |
| 예약 페이지 관리 | 공개 상태, 미리보기, 공유 링크/QR 확인 |
| 예약 상품 관리 | 상품 생성/수정/삭제, 가격, 예약금, 취소 정책, 노출 여부 |
| 예약 운영 | 예약 리스트, 캘린더, 예약 상세, 수동/전화 예약 등록 |
| 상태 처리 | 예약 변경, 취소, 방문 완료, 노쇼 처리 |
| 결제/환불 확인 | 결제 내역, 환불 내역, 수동 처리 상태 확인 |
| 고객 관리 | 고객 프로필, 이력, 메모, VIP/주의 고객 표시 |
| 운영 통계 | 예약/방문/취소/노쇼/예약금/환불 지표 확인 |

## 4. 단계별 개발 계획

### 4.1 Step 0. 프로젝트 기반 구축

목표: 운영 SPA 개발과 테스트를 위한 기본 구조를 만든다.

작업 범위:

- `apps/business-web` Vite React 프로젝트 생성
- TypeScript strict mode 설정
- Tailwind CSS, shadcn/ui, lucide-react 구성
- TanStack Query provider 구성
- Zustand store 기준 정리
- React Router 또는 TanStack Router 라우팅 구성
- OpenAPI generated client 연결
- 인증 보호 라우트와 레이아웃 구성
- 공통 테이블, 폼, 모달, 토스트, 확인 다이얼로그 패턴 구성

산출물:

- 운영 앱 골격
- 공통 Shell layout
- API error boundary
- 공통 상태/폼/테이블 컴포넌트

완료 기준:

- 로그인 전/후 라우팅이 분리된다.
- `pnpm dev`, `pnpm build`, `pnpm test`가 동작한다.

### 4.2 Step 1. 계정과 매장 입점

연계 Phase: Phase 1. 매장 입점/예약 페이지

목표: 오너가 계정을 만들고 매장 입점 신청을 제출할 수 있게 한다.

작업 범위:

- 로그인
- 비밀번호 재설정
- 매장 기본 정보 입력
- 사업자 정보 입력
- 사업자등록증 업로드
- 담당자 연락처 입력
- 승인 요청 제출
- 승인 상태 화면
- 반려 사유 표시와 재제출

화면:

- 로그인
- 비밀번호 재설정
- 입점 신청 시작
- 매장 기본 정보
- 사업자 정보
- 서류 업로드
- 제출 확인
- 승인 상태

API 의존성:

- `POST /api/business/auth/login`
- `POST /api/business/auth/logout`
- `GET /api/business/me`
- `POST /api/business/restaurant-applications`
- `GET /api/business/restaurant-applications/current`
- `PUT /api/business/restaurant-applications/{applicationId}`
- `POST /api/business/files`

완료 기준:

- 오너가 입점 신청을 제출할 수 있다.
- 승인/반려/작성중 상태가 명확히 표시된다.
- 반려된 신청은 수정 후 다시 제출할 수 있다.

### 4.3 Step 2. 매장 설정과 예약 페이지 관리

연계 Phase: Phase 1

목표: 승인된 매장이 예약을 받을 기본 환경을 설정한다.

작업 범위:

- 매장 정보 수정
- 대표 이미지 관리
- 영업시간 설정
- 브레이크타임 설정
- 정기 휴무 설정
- 임시 휴무 설정
- 예약 오픈 기간 설정
- 예약 마감 시간 설정
- 최소/최대 예약 인원 설정
- 예약 페이지 공개/비공개 전환
- 예약 페이지 미리보기
- 공유 링크/QR 표시

화면:

- 매장 설정
- 영업시간 설정
- 휴무 관리
- 예약 정책 설정
- 예약 페이지 관리

완료 기준:

- 매장 오너가 예약 페이지 공개 전 필요한 설정을 완료할 수 있다.
- 공개 전 필수 누락 항목을 화면에서 확인할 수 있다.
- 공개 링크와 미리보기가 제공된다.

### 4.4 Step 3. 예약 상품 관리

연계 Phase: Phase 2, Phase 4

목표: 고객이 선택할 예약 상품을 오너가 구성할 수 있게 한다.

작업 범위:

- 상품 목록
- 상품 생성/수정/삭제
- 상품명, 설명, 가격
- 최소/최대 인원
- 예약 가능 요일/시간
- 재고 수량
- 노출 여부
- 예약금/선결제/무료/현장 결제 정책
- 취소 정책 연결
- 상품 미리보기

제외:

- 단체 예약 상품
- 대관/상담 기반 상품

완료 기준:

- 예약 페이지에 노출할 상품을 생성할 수 있다.
- 상품별 인원/시간/노출/결제 정책을 저장할 수 있다.
- 잘못된 정책 조합은 저장 전에 차단된다.

### 4.5 Step 4. 예약 운영

연계 Phase: Phase 3. 오너 예약 관리

목표: 매장 오너가 일상적으로 예약을 확인하고 상태를 처리할 수 있게 한다.

작업 범위:

- 일별 예약 리스트
- 주/월 캘린더
- 예약 상세 패널
- 예약 상태 배지
- 고객 정보 표시
- 요청사항 표시
- 내부 예약/고객 검색
- 필터: 날짜, 상태, 상품, 시간대
- 수동 예약 등록
- 전화 예약 등록
- 예약 시간/인원 변경
- 매장 취소
- 방문 완료 처리
- 노쇼 처리
- 운영 메모

화면:

- 예약 대시보드
- 일별 리스트
- 캘린더
- 예약 상세
- 수동 예약 등록 모달
- 예약 변경 모달
- 취소/노쇼 확인 모달

완료 기준:

- 오너가 당일 예약을 빠르게 파악할 수 있다.
- 예약 상태 전이가 허용된 범위에서만 수행된다.
- 모든 주요 변경은 감사 로그를 남길 수 있도록 API를 호출한다.

### 4.6 Step 5. 결제/환불 운영 화면

연계 Phase: Phase 4. 예약금/환불

목표: 매장 오너가 예약금, 선결제, 보증, 환불 상태를 확인하고 운영 대응할 수 있게 한다.

작업 범위:

- 결제 내역 목록
- 환불 내역 목록
- 예약 상세의 결제 상태 표시
- 취소 시 환불 예상 금액 표시
- 매장 취소 시 전액 환불 안내
- 환불 실패/처리중 상태 표시
- 플랫폼 관리자 문의가 필요한 상태 표시

제외:

- 자동 정산금 계산
- 정산 지급 스케줄
- 세금계산서 자동 발행

완료 기준:

- 오너가 결제/환불 상태를 예약과 연결해서 확인할 수 있다.
- 환불이 필요한 취소 처리에서 예상 금액을 확인할 수 있다.
- 정산 자동화로 오해될 수 있는 표현을 사용하지 않는다.

### 4.7 Step 6. 좌석/재고 관리

연계 Phase: Phase 5. 좌석/재고 관리

목표: 매장이 시간대별 수용량을 정교하게 통제할 수 있게 한다.

작업 범위:

- 테이블 목록
- 테이블 생성/수정/비활성화
- 좌석 유형 관리: 홀, 룸, 바 등
- 수용 인원 설정
- 상품별 좌석 연결
- 기본 이용 시간 설정
- 타임슬롯 재고 조회
- 임시 마감 처리
- 중복 예약 방지 상태 표시

제외:

- POS 연동
- 대형 단체 예약
- 복잡한 자동 테이블 최적화

완료 기준:

- 오너가 좌석과 상품을 연결할 수 있다.
- 특정 시간대를 임시 마감할 수 있다.
- 재고 계산 결과가 예약 가능 시간에 반영된다.

### 4.8 Step 7. 고객 관리/CRM

연계 Phase: Phase 6. 고객 관리/CRM

목표: 매장 오너가 예약 고객의 방문 이력과 리스크 정보를 관리할 수 있게 한다.

작업 범위:

- 고객 목록
- 고객 상세
- 예약 이력
- 방문 이력
- 노쇼 이력
- 요청사항
- 알레르기
- 기념일
- 매장 메모
- VIP 표시
- 주의 고객 표시
- 차단 고객 범위 표시
- 중복 고객 후보 표시와 병합 요청

개인정보 기준:

- 고객 전화번호는 필요한 화면에서만 표시한다.
- 고객 메모 수정/삭제는 감사 로그 대상이다.
- 개인정보 삭제/익명화 요청 화면을 제공한다.

완료 기준:

- 오너가 고객 이력을 예약 운영에 활용할 수 있다.
- VIP/주의 고객 표시가 예약 상세에 반영된다.
- 개인정보 접근과 다운로드는 제한된다.

### 4.9 Step 8. 운영 통계

연계 Phase: Phase 7. 운영 통계

목표: 매장 오너가 예약 성과를 기간별로 파악할 수 있게 한다.

작업 범위:

- 통계 대시보드
- 기간 필터
- 예약 수
- 방문 완료 수
- 취소 수
- 노쇼 수
- 예약금 매출
- 환불 금액
- 시간대별 예약률
- 상품별 성과
- CSV 내보내기 요청

제외:

- 멀티지점 비교
- 정산 자동화 리포트
- 광고/쿠폰 성과

완료 기준:

- 오너가 단일 매장 기준 핵심 지표를 조회할 수 있다.
- CSV 내보내기가 감사 로그와 함께 동작한다.
- 통계 데이터 기준일과 최근 갱신 시각이 표시된다.

## 8. MVP 완료 기준

- 오너가 입점 신청을 제출하고 승인 상태를 확인할 수 있다.
- 오너가 매장 설정과 예약 상품을 구성할 수 있다.
- 오너가 예약을 조회, 변경, 취소, 방문 완료, 노쇼 처리할 수 있다.
- 오너가 예약금/환불 상태를 확인할 수 있다.
- 직원 권한, 멀티지점, POS, 정산 자동화 없이 단일 매장 운영이 가능하다.
