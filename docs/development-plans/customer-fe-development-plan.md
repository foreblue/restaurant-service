# 사용자 FE 개발 계획

## 1. 개요

사용자 FE는 고객이 식당에서 공유한 예약 링크에 진입해 예약 상품을 확인하고, 날짜/시간/인원 선택, 고객 정보 입력, 결제 또는 보증, 예약 조회/취소까지 수행하는 공개 웹 모듈이다.

기술 기준은 `apps/customer-web`의 Next.js App Router, React, TypeScript strict mode, Tailwind CSS, TanStack Query, React Hook Form, Zod, OpenAPI 기반 API client를 따른다.

사용자 FE는 맛집 탐색, 지도 검색, 통합 검색, 웨이팅을 제공하지 않는다. 진입점은 매장이 공유한 예약 링크, QR, SNS, 홈페이지, 네이버 플레이스 링크 등 외부 채널이다.

## 2. 개발 목표

- 고객이 모바일에서 빠르게 예약을 완료할 수 있는 예약 플로우를 만든다.
- 공개 예약 페이지에 SSR, Open Graph, 기본 SEO 메타데이터를 제공한다.
- 예약 가능 여부, 예약 생성, 취소, 결제/환불 상태를 백엔드 API와 일관되게 표시한다.
- 고객 회원가입 없이 예약번호, 휴대폰 인증 또는 예약 조회 토큰 기반 조회/취소를 지원한다.
- 결제 SDK는 필요한 단계에서만 lazy load한다.

## 3. 모듈 책임

| 영역 | 책임 |
| --- | --- |
| 공개 예약 페이지 | 매장 정보, 대표 이미지, 예약 상품, 예약 정책 표시 |
| 예약 생성 | 상품, 날짜, 시간, 인원, 고객 정보, 요청사항 입력 |
| 예약 확인 | 예약 완료, 예약 번호/상태, 방문 안내 표시 |
| 예약 조회/취소 | 예약 조회 인증, 상세 조회, 취소 가능 여부와 환불 예상 정보 표시 |
| 결제/보증 | 예약금, 전액 선결제, 카드 보증 UI와 결제 결과 처리 |
| 동의/개인정보 | 개인정보 수집 동의, 선택 정보, 마케팅 수신 동의 표시 |

## 4. 단계별 개발 계획

### 4.1 Step 0. 프로젝트 기반 구축

목표: 사용자 FE 앱을 독립적으로 개발, 빌드, 테스트할 수 있는 기반을 만든다.

작업 범위:

- `apps/customer-web` Next.js App Router 프로젝트 생성
- TypeScript strict mode 설정
- Tailwind CSS, ESLint, Prettier 설정
- TanStack Query provider 구성
- React Hook Form, Zod 기반 폼 검증 패턴 정리
- OpenAPI generated client 연결 구조 준비
- 공통 레이아웃, 에러 페이지, 로딩 UI, 빈 상태 UI 구성
- 환경 변수 스키마 정의

산출물:

- Next.js 앱 골격
- 공통 API client wrapper
- 공통 폼/검증 유틸
- 기본 테스트 설정

완료 기준:

- `pnpm dev`, `pnpm build`, `pnpm test`가 동작한다.
- API base URL, 환경 구분, 에러 처리 흐름이 정의되어 있다.

### 4.2 Step 1. 공개 예약 페이지

연계 Phase: Phase 1. 매장 입점/예약 페이지

목표: 승인되고 공개된 매장의 예약 페이지를 외부 링크로 볼 수 있게 한다.

작업 범위:

- `/r/[slug]` 또는 `/reserve/[restaurantId]` 라우트 구현
- 매장 공개 정보 조회
- 매장명, 대표 이미지, 소개, 주소, 전화번호 표시
- 예약 페이지 공개/비공개/승인 대기/운영 정지 상태 처리
- 예약 상품 영역 placeholder 구성
- Open Graph, title, description 메타데이터 구성
- 공유 링크 접근 시 모바일 우선 레이아웃 제공

제외:

- 지도 임베드
- 주변 식당 추천
- 통합 검색 진입

API 의존성:

- `GET /api/public/restaurants/{slug}`
- `GET /api/public/restaurants/{restaurantId}/reservation-page`

완료 기준:

- 공개 상태 매장은 예약 페이지가 표시된다.
- 비공개 또는 승인 전 매장은 예약 불가 안내를 표시한다.
- 링크 공유 시 기본 미리보기 정보가 표시된다.

### 4.3 Step 2. 기본 예약 플로우

연계 Phase: Phase 2. 기본 예약

목표: 고객이 결제 없이 예약을 생성하고 예약 확정/취소 흐름을 경험할 수 있게 한다.

작업 범위:

- 예약 상품 목록 조회 및 선택
- 날짜 선택 UI
- 인원 선택 UI
- 예약 가능 시간 조회 및 선택
- 고객 정보 입력 폼
- 요청사항 입력
- 개인정보 수집 동의
- 예약 생성 요청
- 예약 완료 페이지
- 예약 상세 조회 페이지
- 예약 취소 요청

폼 검증:

- 이름 필수
- 휴대폰 번호 필수
- 인원은 상품 최소/최대 인원 범위 내
- 날짜와 시간은 API가 반환한 가능 옵션만 허용
- 필수 동의 체크 필요

API 의존성:

- `GET /api/public/restaurants/{restaurantId}/reservation-products`
- `GET /api/public/restaurants/{restaurantId}/availability/dates`
- `GET /api/public/restaurants/{restaurantId}/availability/times`
- `POST /api/public/reservations`
- `GET /api/public/reservations/{reservationId}`
- `POST /api/public/reservations/{reservationId}/cancel`

완료 기준:

- 고객이 공개 예약 페이지에서 예약을 완료할 수 있다.
- 예약 불가 시간은 선택할 수 없다.
- 예약 생성 실패 시 명확한 복구 액션을 제공한다.
- 예약 완료 후 예약번호와 상태를 확인할 수 있다.

### 4.4 Step 3. 예약 조회와 취소 UX 강화

연계 Phase: Phase 2, Phase 3

목표: 고객이 예약 이후 자신의 예약을 다시 확인하고 취소할 수 있게 한다.

작업 범위:

- 예약 조회 진입 페이지
- 예약번호 + 휴대폰 인증 또는 조회 토큰 기반 조회
- 예약 상태별 UI
- 취소 가능 시간 표시
- 취소 불가 사유 표시
- 예약 변경은 MVP에서 제한하고, 변경이 필요한 경우 취소 후 재예약 안내

상태별 표시:

| 상태 | 화면 처리 |
| --- | --- |
| `pending` | 예약 접수/확정 대기 표시 |
| `confirmed` | 예약 확정 및 방문 정보 표시 |
| `modified` | 변경된 예약 정보 강조 |
| `cancelled_by_customer` | 고객 취소 완료 표시 |
| `cancelled_by_restaurant` | 매장 취소 안내 표시 |
| `completed` | 방문 완료 표시 |
| `no_show` | 노쇼 처리 안내 |

완료 기준:

- 고객이 예약 상세와 상태를 조회할 수 있다.
- 취소 가능 예약은 취소할 수 있다.
- 취소 불가 예약은 이유를 표시한다.

### 4.5 Step 4. 예약금/선결제/카드 보증

연계 Phase: Phase 4. 예약금/환불

목표: 예약 생성 과정에 결제와 보증 흐름을 연결한다.

작업 범위:

- 상품별 결제 정책 표시
- 예약금 결제 UI
- 전액 선결제 UI
- 카드 보증 등록 UI
- 무료 예약/현장 결제 분기
- 결제 SDK lazy load
- 결제 성공/실패/취소/만료 처리
- 환불 예상 금액 표시
- 결제/환불 상태 표시

API 의존성:

- `POST /api/public/reservations/{reservationId}/payments`
- `GET /api/public/reservations/{reservationId}/payment-summary`
- `POST /api/public/reservations/{reservationId}/guarantee`
- `GET /api/public/reservations/{reservationId}/refund-preview`

정책 결정 필요:

- 예약 생성 전 결제인지, 예약 생성 후 결제 완료 시 확정인지 확정해야 한다.
- 결제 실패 시 임시 예약을 몇 분 동안 보관할지 결정해야 한다.

완료 기준:

- 예약금/선결제 상품은 결제 완료 후 확정 상태로 전환된다.
- 결제 실패 시 고객이 다시 시도하거나 예약을 포기할 수 있다.
- 고객 취소 시 환불 예상 정보가 표시된다.

### 4.6 Step 5. 좌석/재고 정보 반영

연계 Phase: Phase 5. 좌석/재고 관리

목표: 정교해진 좌석/타임슬롯 재고 계산 결과를 예약 UI에 반영한다.

작업 범위:

- 좌석 유형 표시: 홀, 룸, 바 등
- 상품별 가능한 좌석 유형 표시
- 시간대별 잔여 수량 표시 여부 정책 반영
- 임시 마감 시간대 비활성화
- 동시 예약 실패 시 즉시 재조회와 대체 시간 제안

제외:

- 고객이 특정 테이블 번호를 직접 선택하는 기능
- 대형 단체/대관 예약

완료 기준:

- 좌석/재고 계산 결과가 고객 선택지에 정확히 반영된다.
- 동시성 충돌 시 고객에게 재선택 흐름을 제공한다.

### 4.7 Step 6. 고객 정보와 CRM 입력 확장

연계 Phase: Phase 6. 고객 관리/CRM

목표: 예약 시 고객 관리에 필요한 선택 정보를 수집하되, 입력 부담을 낮춘다.

작업 범위:

- 알레르기 입력
- 기념일 선택/입력
- 요청사항 템플릿
- 마케팅 수신 동의
- 개인정보 수집/이용 동의 문구 관리
- 민감 정보 입력 주의 안내

완료 기준:

- 고객이 선택 정보를 입력할 수 있다.
- 입력 정보가 예약 상세와 고객 프로필로 전달된다.
- 선택 정보 미입력 시에도 예약 흐름이 막히지 않는다.

### 4.8 Step 7. 품질 안정화와 운영 준비

연계 Phase: 전체

목표: 공개 예약 플로우의 안정성과 운영 관측성을 확보한다.

작업 범위:

- 예약 생성 E2E 테스트
- 예약 취소 E2E 테스트
- 결제 성공/실패 E2E 테스트
- 주요 오류 client-side reporting 연결
- 접근성 점검
- 모바일 뷰포트 QA
- 성능 측정: LCP, CLS, API 응답 지연 표시

완료 기준:

- 모바일 기준 핵심 예약 플로우가 깨지지 않는다.
- 주요 API 실패 케이스에 사용자 복구 경로가 있다.
- 릴리스 전 E2E 테스트가 자동 실행된다.

## 8. MVP 완료 기준

- 고객이 공개 예약 페이지에서 예약을 생성할 수 있다.
- 고객이 예약을 조회하고 취소할 수 있다.
- 결제 또는 보증이 필요한 상품의 예약 흐름이 동작한다.
- 웨이팅, 지도, 통합 검색 없이 외부 링크 기반 예약 경험이 완성된다.
